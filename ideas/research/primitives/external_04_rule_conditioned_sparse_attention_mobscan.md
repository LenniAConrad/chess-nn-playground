# Five Candidate Novel Neural-Network Primitives for Chess Position Evaluation

**TL;DR**
- The five proposals that survived self-audit are **Ray-Cast Sparse Attention (RCSA)**, **Mobility-Scan SSM (MobScan)**, **Edit-Differential Accumulator (EDA)**, **Signed-Permutation Involution Layer (SPIL)**, and **Content-Routed Hypergraph Block Attention (CRHBA)** — three of the five (RCSA, MobScan, CRHBA) live in the content-dependent / input-determined-connectivity vein the user asked to lean into, with EDA covering the HalfKA-style O(Δ)-incremental axis and SPIL covering the chess-group-equivariance axis.
- Only **two** are plausibly *fully* novel as torch.nn-style primitives (RCSA and EDA); the other three are best framed as *underexplored-for-chess specialisations* of an existing thread of literature (MobScan ↔ Mamba-2/Graph-Mamba; SPIL ↔ Agrawal–Sicilia–Marsiske signed-perm-rep G-DNNs; CRHBA ↔ MoBA/NSA), and the report surfaces that prior art explicitly per the user's instruction.
- A single primitive that *purely* meets the bar of "does not decompose into existing PyTorch ops AND is chess-relevant AND is O(Δ)-friendly" is rare; the best bet for engine impact is EDA + RCSA together — EDA is the speed win at MCTS-call time, RCSA is the structural-novelty bet for tactical discrimination.

## Key Findings

1. **The richest novelty vein is "graph generated from the input, not from learned attention scores".** Existing dynamic-graph work (Saqur et al. 2024 DGG, Dong et al. 2024 DGCN, AlphaGateau GATEAU layer, Alwer 2023 chess-GNN) still learns or fixes a graph through learnable attention; none of these match the chess situation where the connectivity is a *deterministic function of integer piece positions* (legal-move graph, attack graph, ray-of-sight graph). A primitive whose graph is materialised by a non-learned, content-dependent indexing function on the input genuinely changes the computation graph relative to masked attention because the mask is not a real-valued logit on a fixed n×n candidate set — it is a sparse integer index tensor recomputed per forward pass.

2. **Selective SSMs (Mamba/Mamba-2/Gated DeltaNet) are the closest thing in the literature to a clean content-dependent recurrence**, and graph-Mamba variants exist, but a scan whose *adjacency itself* is determined by the input (rather than the scan parameters being input-dependent on a fixed sequence) is unexplored. This is the MobScan slot.

3. **Differential Transformer (Ye et al. 2024) and Diff-Attention** look novel but on close inspection decompose into two parallel softmax-attention calls followed by a learned-scalar subtraction — i.e., a composition of existing PyTorch ops. It is cut from the list for that reason (see "What I cut").

4. **The HalfKA O(Δ) property generalises to a primitive**, but only if the layer is mathematically linear in its sparse input *and* its non-linearity is applied after accumulation. NNUE exploits exactly this. EDA tries to lift this to a *general* primitive with a typed (state, delta) → state signature and a contract that any non-linearity is post-accumulator clip-ReLU. No existing torch.nn op carries this contract; nn.EmbeddingBag is the closest and does not expose the incremental form.

5. **The chess symmetry group is larger than D4.** Side-to-move involution σ is a Z/2 action that acts simultaneously as (board-flip, color-swap, sign-flip on evaluation, optional sign-flip on feature channels). Agrawal et al. 2023 (Densely Connected G-invariant DNNs with Signed Permutation Representations) supply exactly the machinery for this; no public chess network uses it. This is an honest reframing, not a new mathematical object.

### primitive_rcsa

**Name:** Ray-Cast Sparse Attention (RCSA)

**One-line claim:** Attention whose key set per query is generated by ray-casting through an input occupancy bitmap, so the adjacency tensor is integer-valued and recomputed every forward pass.

**Mathematical signature:** `RCSA: R^{[B,64,d]} × {0,1}^{[B,64]} → R^{[B,64,d]}`. Given features X∈R^{B,64,d} and occupancy O∈{0,1}^{B,64}, deterministically compute idx∈Z^{B,64,K} where idx[b,q,k] is the square reached by ray k (eight ray directions × up to ⌈log₂64⌉ stops) cast from q until hitting an occupied square in O[b,·]. Then `Y[b,q] = Σ_k softmax_k(⟨W_q X[b,q], W_k X[b,idx[b,q,k]]⟩ / √d) · W_v X[b,idx[b,q,k]]`. K is fixed (≤8 rays · ≤7 stops = 56, in practice ≤16 used). Gradient flows through X via the standard scaled-dot path; idx is treated as a non-differentiable lookup (no gradient through occupancy, which is fine because O is derived from the *input encoding* not from learned parameters).

**Why this does not decompose into existing PyTorch ops:** The closest op is masked `scaled_dot_product_attention` with `attn_mask=M`. The decisive difference is that an n×n masked attention has fixed shape O(n²) candidate logits and zeroes the unwanted ones; RCSA *never materialises* a 64×64 logit matrix because idx is a sparse integer-index tensor of shape [B,64,K] with K≪64 computed by a custom CUDA/Triton kernel on the input bitmap. The forward pass is gather→bmm→softmax→gather, not (Q@Kᵀ)→mask→softmax. The connectivity is content-dependent in the strict sense required by the task definition: the *set of indices participating in the computation* is a function of the input data, not of fixed positions or learned scores.

**Chess-specific motivation:** Sliding-piece tactics (pins, skewers, X-rays, discovered attacks) are defined by line-of-sight through occupancy. A residual conv stack must rediscover ray semantics from scratch every training run; RCSA hands the network the actual ray topology as a hard inductive bias while still letting the *content* of each ray be learned. This directly targets the CRTK class-1 hard-negative cases where one piece interposes and breaks a pin.

**Generalisation beyond chess:** Ray-casting attention generalises to any line-of-sight problem: 2D/3D occupancy grids in robotics (visibility graphs), differentiable rendering (primary-ray queries), and astrophysics radiative-transfer surrogates.

**Complexity (forward, backward, incremental-update):**
- Forward: O(B·64·K·d) ≈ O(B·n·K·d), vs. masked attention O(B·n²·d).
- Backward: O(B·n·K·d) — same as forward up to constants; the gather is auto-diff-friendly.
- Incremental update on bounded-change input: O(rays-touching-changed-squares · K · d) ≈ O(K·d) when only one piece moves; the idx tensor only needs recomputation for rays whose stop-set includes a changed square (≤16 squares per move).

**Scout-scale falsification test:** Drop RCSA into the `i243` HalfKA+dual-stream proposal, replacing one of its self-attention blocks with RCSA (same d, same heads). Train 12 epochs on the 173k-position corpus, single 3070, single seed, ≤2 GPU-hours. Metric: matched-recall near-puzzle FP rate on the CRTK class-1 verified-near-puzzles set. **Success** = strictly lower FP rate than the unmodified i243 *and* wall-clock per node within 1.3× of i243. **Failure** = either condition violated.

**Failure mode catalogue:**
- (a) Hidden rebrand risk: a reviewer will argue this is just "masked attention with a precomputed mask". Counter: the mask is never materialised; only the K-sparse index tensor is. The op also has fundamentally different complexity class (O(n·K) vs O(n²)).
- (b) Numerical instability: when K_actual < K_max (rays terminate early), padded entries need a masked softmax; sloppy padding produces NaNs at empty positions.
- (c) Too slow to be useful: the ray-cast kernel is the bottleneck. On a 3070 a naive PyTorch implementation will run 5–10× slower than the matmul-only baseline; only a fused Triton kernel makes the FLOPs win real. Honest expectation: useful only with a Triton kernel.

**Prior art surfaced:** Closest dynamic-connectivity work is **Saqur et al., "Learning Adaptive Neighborhoods for Graph Neural Networks", arXiv:2307.09065** (differentiable graph generation), **Dong et al., "Differentiable Cluster Graph Neural Network", arXiv:2405.16185** (input-induced cluster edges), and **Alwer 2023 / Rigaux et al. 2024 "Enhancing Chess RL with Graph Representation" (AlphaGateau), arXiv:2410.23753** (chess-as-graph with mobility edges). None cast rays through occupancy; AlphaGateau uses the legal-move multigraph but with a learned GAT, not a ray-determined sparse attention. **Native Sparse Attention (Yuan et al. 2025, arXiv:2502.11089)** and **MoBA (Lu et al. 2025, arXiv:2502.13189)** also do content-dependent sparsity, but their blocks are *learned* via gating/top-k on similarity, not derived from a non-learned input rule.

**Status:** proposed

### primitive_mobscan

**Name:** Mobility-Scan Selective SSM (MobScan)

**One-line claim:** A Mamba-2-style selective state-space scan whose scan order and edge set are the legal-move graph of the input position, not a 1-D sequence.

**Mathematical signature:** `MobScan: R^{[B,64,d]} × Adj^{[B,64,64]} → R^{[B,64,d]}`, where Adj is the (input-determined) sparse legal-move adjacency. The recurrence on the directed acyclic ordering of squares-by-rank is: `h_s = A_s(X) h_{parents(s)} + B_s(X) X_s`, `Y_s = C_s(X) h_s`, with A_s, B_s, C_s diagonal data-dependent (Mamba-style). Aggregation over parents uses a max-plus or summation associative reduction; both admit a parallel chunkwise scan via the state-space-duality (SSD) formulation. Gradient: standard SSM gradient with parents instead of t−1.

**Why this does not decompose into existing PyTorch ops:** The closest op is `mamba_ssm.selective_scan` (or Mamba-2's `ssd`), which assumes a *fixed 1-D causal* parent relation t→t-1. MobScan replaces the implicit linear chain with an input-determined sparse DAG, which means the scan kernel cannot be a precompiled chunked-causal kernel; it requires per-batch graph compilation. No `torch.nn` operator scans over input-dependent DAGs.

**Chess-specific motivation:** Information in chess propagates along legal moves and attacks, not along a raster scan. A piece's tactical role is its position in the mobility graph, which differs per position. Mamba-style selective decay is already the right inductive bias for "remember important squares, forget irrelevant ones"; binding the scan to the actual mobility DAG removes the need for the network to learn the chain rule of attacks from data.

**Generalisation beyond chess:** Any combinatorial-graph problem with input-dependent topology where local "flow of influence" matters: dynamic routing, traffic networks, dataflow graphs in program analysis, reaction networks in chemistry.

**Complexity (forward, backward, incremental-update):**
- Forward: O(B · |E| · d) where |E| is the number of legal moves (≈30–60 average), vs. full attention O(n²d)=O(4096·d).
- Backward: O(B · |E| · d) (reverse scan).
- Incremental update on bounded-change input: O(|E_changed| · d). When one piece moves, only the squares whose mobility changed (typically <10) need re-scanning; the rest of h is unchanged. This gives a HalfKA-flavoured property for the recurrent state.

**Scout-scale falsification test:** Replace the middle conv stack in `i193` (the best conv-only baseline) with two MobScan layers of equal parameter count. Train 12 epochs on 173k positions. Metric: matched-recall near-puzzle FP rate. **Success** = match or beat i193 FP rate at ≤1.2× wall clock. **Failure** = worse FP rate or >1.5× wall clock.

**Failure mode catalogue:**
- (a) Hidden rebrand risk: looks like "Graph-Mamba" (Behrouz–Hashemi 2024 et seq.). Honest answer: it *is* graph-Mamba, restricted to input-determined graphs. The novelty is that the graph is a *deterministic non-learned function of the input*, which neither Graph-Mamba nor any GNN-Mamba hybrid currently does.
- (b) Numerical instability: state-decay parameters A_s near 1 plus long mobility chains (queen rays) produce gradient explosion; needs the same complex-eigenvalue stabilisation Mamba-2 uses.
- (c) Too slow to be useful: per-batch graph compilation kills throughput unless graphs are batched by topology — they will not be. Need a Triton kernel that does dynamic-shape scans.

**Prior art surfaced:** **Gu & Dao 2023 (Mamba, arXiv:2312.00752)**, **Dao & Gu 2024 (Mamba-2 / SSD, arXiv:2405.21060)**, **Yang et al. 2024 (Gated DeltaNet, arXiv:2412.06464, ICLR 2025)** all give the building-block recurrence; **Wang et al. 2024 (Graph-Mamba)** and **Behrouz–Hashemi 2024** apply Mamba on a fixed graph; none use an input-determined graph. **AlphaGateau (Rigaux 2024)** uses input-determined graphs but with GAT, not selective SSM. This proposal is a deliberate cross-product of those two streams. Reframe honestly as "underexplored for chess".

**Status:** proposed

### primitive_eda

**Name:** Edit-Differential Accumulator (EDA)

**One-line claim:** A primitive whose forward signature accepts (previous state, sparse edit) and outputs the new state, with a mathematical contract that the result equals the full forward on the edited input.

**Mathematical signature:** `EDA.forward(x: SparseInt, edit: list[(idx_remove:int, idx_add:int)] or None, state: R^{[B,m]}) → R^{[B,m]}`. Implements `state' = state + Σ_a E[idx_add_a] − Σ_r E[idx_remove_r]` where `E ∈ R^{N,m}` is a learned embedding matrix. The *contract* (enforced by `eda.full_forward(x) ≡ eda.full_forward(x_prev) + delta(edit)`) requires that any non-linearity must be applied **after** the accumulation (clip-ReLU is the canonical choice, as in NNUE), and the primitive exposes both forms (`full_forward` and `incremental_forward`) sharing identical weights. Backward through `full_forward` is standard; backward through `incremental_forward` is gradient-stopped at `state` and flows only through the added/removed embedding rows.

**Why this does not decompose into existing PyTorch ops:** The closest is `nn.EmbeddingBag(mode='sum')`. EmbeddingBag does not expose a `(state, delta) → state'` API and offers no formal equality guarantee between an incremental call and a full call; its kernel always re-reduces the full bag. EDA is a *primitive with two coupled forward kernels and a typed state object* — the state is a first-class output of the layer, not an internal hidden activation. The dual-form contract changes the computation graph: the autograd graph for `incremental_forward` contains only the touched embedding rows, while a re-run of `EmbeddingBag(x_new)` produces a graph touching every active index in x_new.

**Chess-specific motivation:** This is the operator NNUE/HalfKA implicitly uses but never reified. Stockfish's update_accumulator routine is hand-coded around a specific feature set. EDA exposes the property as a *layer*: any model that wants O(piece-moves) inference cost can compose EDA layers as its first stage, get the speed win automatically, and remain trainable end-to-end in the full-forward form.

**Generalisation beyond chess:** Streaming recommender systems with sparse user-item edits; incremental token-edit re-encoding in interactive code/text editors; incremental scene graphs in robotics where only one object moves per frame.

**Complexity (forward, backward, incremental-update):**
- Forward (full): O(B · k · m), k = active features per sample. Equal to EmbeddingBag.
- Backward: O(B · k · m).
- Incremental update on bounded-change input: **O(|edit| · m)** with a constant of typically 4 (one piece move = remove 1–2, add 1–2 features in HalfKA). This is the property that the user explicitly chases.

**Scout-scale falsification test:** Replace the i243 input embedding stage with EDA. Train 12 epochs on 173k positions, single 3070. Metric: (1) matched-recall FP rate on CRTK class-1 must be within noise of i243; (2) **wall-clock per MCTS-node-call** measured with synthetic 10k-position playout must be ≥2× faster than i243. **Success** = both. **Failure** = either fails.

**Failure mode catalogue:**
- (a) Hidden rebrand risk: "It's just EmbeddingBag with a delta call." Counter: the *contract* (mathematical equality of two forward paths + a typed state) is what is novel. EmbeddingBag exposes no such object.
- (b) Numerical instability: if downstream layers are non-linear before the accumulator (e.g., a LayerNorm), the equality breaks. EDA must refuse to compose with non-pointwise pre-accumulation ops; this is a usability constraint, not a math one.
- (c) Too slow to be useful: EDA's full-forward path is identical in cost to EmbeddingBag, so the only payoff is the incremental path, which only matters at engine-inference time. If chess-NN-playground's main metric is training-time FP rate, EDA is invisible. The win is engine integration, not training metrics.

**Prior art surfaced:** **NNUE (Nasu 2018; Stockfish-NNUE 2020)** is the canonical instance; **HalfKA / HalfKAv2** (Stockfish 12+) is the dominant feature set; **Liefooghe 2021 "Stockfish NNUE accumulator"** technical note documents the in-place update. Academic generalisations are essentially absent — no torch.nn.IncrementalEmbeddingBag exists. The closest streaming-net work is event-camera SNN literature (Gehrig–Scaramuzza 2024) but their primitives are spike-driven, not delta-accumulator-style. This is the most plausibly *truly new* primitive in the list.

**Status:** proposed

### primitive_spil

**Name:** Signed-Permutation Involution Layer (SPIL)

**One-line claim:** A linear layer with weights constrained to commute with the chess side-to-move involution σ (board-flip × color-swap × sign-flip), implemented via signed-permutation representations.

**Mathematical signature:** `SPIL: R^{[B,8,8,c]} → R^{[B,8,8,c']}` with weight tensor W ∈ R^{c×c'×k×k} constrained to satisfy `σ ∘ W = W ∘ σ` where σ acts as (vertical board flip) ⊗ (sign-flip on a chosen subset S ⊂ {1..c} of channels — the "color-anti-symmetric" channels) ⊗ (sign-flip on a subset S' ⊂ {1..c'} of outputs). Implementation: W is parameterised by an unconstrained tensor U; the actual W = (U + Pσ U Pσ⁻¹) / 2 where Pσ is the signed-permutation representation of σ. Gradient flows through U.

**Why this does not decompose into existing PyTorch ops:** A normal `nn.Conv2d` does not enforce the joint sign-flip / spatial-flip equivariance; weight-sharing schemes like `e2cnn` enforce equivariance for D4 (rotations + reflections, no sign-flip on features). The novel piece is the *signed* permutation representation acting on feature channels in addition to spatial flip — this is a representation-theoretic object that does not appear in any released torch.nn extension. The constraint changes the *parameter manifold*, not just the forward formula.

**Chess-specific motivation:** Side-to-move flip is a perfect Z/2 symmetry in chess: evaluating from White's side at position p and from Black's side at the flipped position σ(p) must give *negated* scores. Standard nets are trained on both views and hope to learn this; SPIL bakes it in. This halves the effective data requirement for symmetry-related positions and guarantees consistency.

**Generalisation beyond chess:** Any task with a paired-input sign-flip invariance: physics simulations with charge-conjugation symmetry; antisymmetric wavefunction networks (already adjacent to this in Pfau et al. 2020 FermiNet); two-player zero-sum games (Go, Othello, Connect-4, Shogi).

**Complexity (forward, backward, incremental-update):**
- Forward: same FLOPs as the corresponding Conv2d once W is materialised; O(2×) cost in weight construction at training time, amortised to zero at inference.
- Backward: O(Conv2d).
- Incremental update on bounded-change input: not applicable in a useful sense (the layer is dense convolution).

**Scout-scale falsification test:** Replace the conv blocks of `i193` with SPIL convs (same #params). Train 12 epochs on 173k positions, single seed. Two metrics: (1) matched-recall FP rate on CRTK class-1 must match or beat i193; (2) test for *exact* sign-flip consistency: feed σ(p) and check that |f(p) + f(σ(p))| < 1e-5 over a held-out set. **Success** = (1) and (2). **Failure** = either fails. ≤2 GPU-hours.

**Failure mode catalogue:**
- (a) Hidden rebrand risk: this is "G-equivariant CNN for an extended group", which Cohen–Welling 2016 et al. already invented; the novel ingredient is the *signed* representation, which Agrawal–Sicilia–Marsiske 2023 already laid down. **Frank assessment: this is a reframing, not a new primitive.** Keep per the user's instruction ("when a 2024–2026 paper already proposes something close: KEEP and reframe as underexplored for chess").
- (b) Numerical instability: the projection (U + PUP⁻¹)/2 is conditioning-neutral but can hide rank loss in U; needs an init that avoids the σ-fixed subspace collapsing.
- (c) Too slow to be useful: zero inference overhead, so this failure mode does not apply.

**Prior art surfaced:** **Cohen & Welling 2016 (G-CNNs, arXiv:1602.07576)**; **Agrawal, Sicilia & Marsiske 2023, "Densely Connected G-invariant DNNs with Signed Permutation Representations", arXiv:2303.04614** (this is the exact mathematical object SPIL uses); **Pfau et al. 2020 (FermiNet)** for antisymmetric two-player analogue. Honest reframing as "underexplored for chess" required by the user's specification.

**Status:** proposed

### primitive_crhba

**Name:** Content-Routed Hypergraph Block Attention (CRHBA)

**One-line claim:** Block attention where blocks are hyperedges generated from the input (piece-coalition or attacker-defender groups), not contiguous token spans, and the router selects among input-generated hyperedges.

**Mathematical signature:** `CRHBA: R^{[B,n,d]} × HE^{[B,H]} → R^{[B,n,d]}`, where HE is a set of H input-determined hyperedges (each a variable-length list of token indices), produced by a fixed rule R(x) (e.g., R returns one hyperedge per (attacker, defender) chain). For each query token q, a learned router scores affinity to each hyperedge h: `s_{q,h} = ⟨W_q X_q, mean_{i∈h} W_k X_i⟩`. Top-k hyperedges are selected; the query attends to the union of their member tokens. Gradient flows through Q/K/V projections; hyperedge membership is non-differentiable (computed from input).

**Why this does not decompose into existing PyTorch ops:** MoBA blocks are fixed contiguous token spans (`token[i*B:(i+1)*B]`); NSA blocks are also contiguous compression / selection windows. Neither has *variable-length, input-determined, possibly overlapping* hyperedges. The closest existing op composition is "compute mask M(x), do masked attention" — but M is not a mask, it is a sparse member-set with overlapping coverage, and the routing score is computed on hyperedge *centroids* rather than individual key tokens. This places it strictly between MoBA and a hypergraph-attention layer; no PyTorch primitive covers this.

**Chess-specific motivation:** Tactical motifs are coalitions: fork = (attacker, two-targets), pin = (attacker, pinned-piece, king), battery = (queen, rook, file). Each is a hyperedge derivable from the board. Letting the network attend to *hyperedges* rather than individual squares matches how human players reason about tactics and concentrates capacity on the units that actually matter for evaluation.

**Generalisation beyond chess:** Multi-agent reinforcement learning (coalitions of agents); molecular property prediction (functional groups as hyperedges); code modelling (def-use chains as hyperedges).

**Complexity (forward, backward, incremental-update):**
- Forward: O(B · n · k · |h̄| · d) where k = selected hyperedges per query, |h̄| = average hyperedge size, k·|h̄| << n. For chess with n=64, k=4, |h̄|≈4: ≈O(64·16·d) per sample vs n²·d = 4096·d for full attention.
- Backward: O(forward).
- Incremental update on bounded-change input: hyperedge set is recomputed; for a one-piece move only the hyperedges containing the moved piece change. O(|changed-hyperedges| · k · |h̄| · d). Not as clean as EDA's O(Δ), but sub-linear in n.

**Scout-scale falsification test:** Replace i242's chess-decomposed-attention block with CRHBA, using the AlphaGateau attack-graph rule to generate hyperedges. Train 12 epochs on 173k positions. Metric: matched-recall FP rate on CRTK class-1 vs. i242 and vs. i243. **Success** = strict improvement over i242 (which underperformed at small scale) and within 10% of i243's wall clock. **Failure** = either fails. Also flag scale: if FP rate is worse than i193's conv-only baseline, this confirms attention-is-data-hungry at 173k and the result is inconclusive on novelty.

**Failure mode catalogue:**
- (a) Hidden rebrand risk: a reviewer will say "this is MoBA with a different block-partition function". Honest counter: MoBA's blocks are non-overlapping contiguous spans defined offline; CRHBA's hyperedges are overlapping, variable-size, and computed per forward pass from the input. The kernel structure (gather-scatter into variable-size groups) is genuinely different from MoBA's block-diagonal flash kernel.
- (b) Numerical instability: variable hyperedge sizes need careful normalisation in the centroid; tiny hyperedges (size 1) cause router collapse onto degenerate experts.
- (c) Too slow to be useful: variable-shape gather kernels are slow on a 3070 without a custom Triton implementation; expected 2–4× slowdown vs naïve attention until kernelised.

**Prior art surfaced:** **MoBA (Lu et al. 2025, arXiv:2502.13189, NeurIPS 2025 spotlight)**; **Native Sparse Attention (Yuan et al. 2025, arXiv:2502.11089)**; **FlashMoBA (Xiao et al. 2025, arXiv:2511.11571)** for the kernel side; **Soft MoE (Puigcerver et al. 2024)** for the routing analogy. Honest framing: this is *MoBA generalised to hypergraph blocks*. Underexplored for chess.

**Status:** proposed

## What I Cut

1. **Differential-Attention layer (Diff Transformer, Ye et al. 2024, arXiv:2410.05258).** Cut because it decomposes into two parallel `softmax(QKᵀ/√d)V` calls and a learned-scalar subtraction — i.e., a composition of existing PyTorch ops with no new gradient topology. It is a useful trick but does not clear the "does not decompose" bar.

2. **TTT-Layer for chess (Sun et al. 2024, arXiv:2407.04620).** Genuinely novel as a primitive (hidden-state = small NN trained at inference time), but cut for chess specifically because (i) it requires non-trivial backward-through-backward gradient computation which is brittle on a 3070, and (ii) chess inference is single-position not streaming — the inner-loop self-supervised objective has no natural target. Mentioned as background only.

3. **Mixture-of-Recursions per-token routing for chess (Bae et al. 2025, arXiv:2507.10524, NeurIPS 2025).** Novel as a primitive (input-dependent depth via a learned router), but cut because (i) chess positions are not token sequences of varying difficulty in a meaningful way at the per-square level, and (ii) MoR's win is throughput on long contexts, which is orthogonal to the 64-square setting. Better used as a building block in a future iteration than proposed as a chess-specific primitive.

4. **"Learned positional encoding for chess squares".** Cut as an encoding tweak, not a primitive — violates the user's hard constraint #2 ("No input-encoding proposals").

5. **Stick-breaking attention / Sigmoid-attention (recent 2024 variants).** Cut as attention-normalisation rebrands — they replace softmax with a different per-row reduction. The forward decomposes into existing pointwise + reduce ops; gradient flow class is unchanged. Anti-example category per the user's spec.

## Recommendations

1. **First experiment to run (≤2 GPU-hours, single 3070):** EDA as the input stage of i243, measuring wall-clock-per-MCTS-call against i243 baseline on a synthetic 10k-position playout. Decision rule: ship to engine integration if speedup ≥2× at no FP-rate regression. Threshold to abandon: <1.3× speedup. EDA is the lowest-risk, highest-engine-payoff bet because the math is essentially proven (NNUE) and the novelty is purely operator-API.

2. **Second experiment (parallel):** RCSA dropped into one i243 attention block. Decision rule: ship if CRTK class-1 matched-recall FP rate is strictly lower than i243 *and* wall-clock within 1.3× *with a Triton kernel*; otherwise note as theoretically interesting but practically blocked on kernel work. Threshold to abandon: FP rate ≥ i243 at 1.3× wall-clock.

3. **Third experiment (if 1 and 2 partial success):** SPIL conv replacement of i193's middle blocks. Decision rule: ship if FP rate ≤ i193 *and* the σ-consistency check is exact (< 1e-5). This is the cheapest win in the list because it adds zero inference cost.

4. **Hold MobScan and CRHBA for after the above settle.** Both require custom Triton kernels to be meaningfully tested at the 173k-position scale; commit kernel-development time only if at least one of the first three shows a measurable signal. Threshold to invest: any of EDA/RCSA/SPIL produces ≥10% relative reduction in CRTK class-1 FP rate or ≥1.5× engine-call speedup.

5. **Frame the eventual paper around EDA + RCSA as the genuinely novel pair**, with SPIL/MobScan/CRHBA positioned as principled adaptations of recent published primitives to chess. Do not over-claim novelty on the latter three.

## Caveats

- I have not run any of these primitives. All performance claims are predictions, explicitly flagged as such per the user's "do not invent empirical results" rule.
- The novelty of EDA hinges on the claim that no torch.nn primitive currently exposes a typed (state, delta) → state API with a coupled full-forward contract. I have surveyed the public torch.nn surface and NNUE literature, but there is non-zero risk that a niche library (e.g., a streaming-recommender package) implements something equivalent. Worth a final targeted literature pass before paper submission.
- RCSA's "does not decompose" argument depends on the ray-cast kernel being implemented as a custom op; a naive PyTorch implementation that materialises a 64×64 sparse mask *does* decompose into masked attention. The primitive's novelty is therefore partly *implementation-defined*, which is a legitimate reviewer objection.
- The user's constraint that "Stockfish scores, PVs, node counts may NOT enter the primitive's compute graph as inputs" is respected throughout: all proposals take only board encodings (occupancy bitmap, piece-square features) as input.
- Mamba-2 / Gated DeltaNet / NSA / MoBA are all 2024–2025 and the field is moving weekly; the prior-art survey is current as of May 2026 but a paper appearing between scout-test and submission could collapse one of the three "reframed" proposals into prior art proper. MobScan and CRHBA carry the highest risk of being scooped before publication.
- The `run_blocking_subagent` and `enrich_draft` tools referenced in the task instructions were not actually present in the tool environment for this session, so the late-stage subagent gap-fill and one-shot enrichment pass were not performed; the draft was written directly from 12 web_search rounds plus the user's principles. A subsequent pass with those tools (if available in a future session) would most usefully target (a) confirming no published `IncrementalEmbeddingBag`-style torch primitive exists, and (b) verifying that AlphaGateau / chess-GNN literature has not already proposed a ray-cast or hyperedge attention.
